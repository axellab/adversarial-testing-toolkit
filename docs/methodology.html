Adversarial Testing Toolkit · Metodología

Romperlo antes
que lo rompa la realidad

Cómo el framework aplica el red-teaming — la disciplina usada para quebrar modelos — a código, sistemas, prompts y planes de implementación.

La idea única: asumir que algo será usado del peor modo plausible y encontrar dónde eso lo rompe — con escenario concreto, severidad y test de regresión. No una opinión; una falla reproducible.

03
El modo de pensar

Técnicas para generar ataques

assumptions

Invertir cada supuesto

“El input viene validado”, “el servicio siempre está arriba”. Listarlos y violar cada uno.

boundaries

Empujar a los bordes

Cero, uno, muchos, vacío, negativo, malformado, duplicado, fuera de orden.

trust

Seguir las fronteras de confianza

Dónde un dato no confiable cruza a un intérprete, tool, BD, modelo u otro usuario.

dependencies

Matar las dependencias

Asumir que todo lo externo está lento, caído o mintiendo. Trazar qué pasa entonces.

time / order

Romper tiempo y orden

¿Y si corre dos veces? ¿A medias? ¿Concurrente? ¿Tras un reinicio? ¿Nunca?

threats

Pensar como cada amenaza

Usuario con datos raros, atacante motivado, bot, insider. Uno por uno.

04
Qué buscar · cómo pesar

Cinco dimensiones, una regla de severidad

FUNCCorrección funcional — bordes, entrada inválida, idempotencia, redondeo de dinero, concurrencia.
SECSeguridad — inyección, authZ/authN, secretos, SSRF, exposición de datos, rate limiting.
RESResiliencia & performance — timeouts, fallo de dependencias, carga, leaks, fallo parcial, observabilidad.
LLMRobustez LLM — inyección de prompt, jailbreaks, exfiltración, abuso de tools, grounding, coste.
PLANSpecs & planes — ambigüedad, supuestos, requisitos faltantes/en conflicto, criterios no testeables.
Severidad = Impacto × Probabilidad · nunca por categoría sola

05
Tres intensidades

La misma disciplina, dosificada

fondo

/adversarial-mode on — persistente

Un hook determinístico anexa la sección “cómo puede fallar esto” a cada respuesta. Adversarial testing como hábito.

focalizado

/redteam <artefacto> — crítico único

Un agente ataca un solo artefacto (plan, spec, código, prompt) y devuelve solo el análisis de fallas.

exhaustivo

/deep-audit <target> — multi-agente

Cuatro auditores en paralelo, un dimensión cada uno, con verificación adversarial antes de sintetizar.

06
El pipeline del deep-audit

Amplitud por fan-out, confianza por el escéptico

1 · Fan-out en paralelo

Los auditores relevantes corren a la vez sobre el mismo target, cada uno cargando el núcleo.

weakpoint-scoutsecurity-auditorresilience-auditorllm-robustness-auditor
2 · Consolidar & deduplicar

Los ids de taxonomía compartidos permiten fusionar el mismo hallazgo visto por dos lentes en uno.

3 · Verificación adversarial EL PASO CLAVE

Cada Critical/High se red-teamea contra sí mismo (“intentá refutarlo: ¿es real? ¿la severidad es correcta?”) para descartar falsos positivos y corregir severidades infladas.

4 · Sintetizar

Un único reporte ordenado por severidad — escenario + fix + test de regresión + blind spots honestos.

5 · Enrutar a canales

El mismo hallazgo, re-renderizado: terminal · archivo · artifact HTML · comentarios de PR.

fuera de los sembrados
La verificación adversarial pagó: refutó el escenario “sin token devuelve SSN” (en realidad da 500; el path real es token basura) y confirmó por ejecución la caché envenenada. Sin ese paso, el reporte habría llevado un escenario incorrecto.
Seguridad — análisis y testing adversarial solamente. Nunca ataques reales, exfiltración ni armado de exploits. El output es un sistema más fuerte, no uno comprometido. Y honestidad de cobertura: lo que no se revisó se declara como blind spot, nunca se implica cobertura que no se hizo.
